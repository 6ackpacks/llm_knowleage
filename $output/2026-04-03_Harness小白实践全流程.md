# Harness 应用与实践全流程：小白入门指南

## 一、先搞清楚你为什么需要 Harness

你有没有遇到过：让 AI 写一段代码很顺，但让它接着干第二件事就开始乱？或者让它跑一个复杂任务，跑到一半它"忘了"最开始的要求？

这不是模型的问题，这是你缺了 **Harness**。

**一句话定义**：Harness 就是给 Agent 搭的操作系统。模型是 CPU，你不能对着裸 CPU 敲键盘。

没有 Harness，无论你用多贵的模型，结果都是：
- 任务稍复杂就跑偏
- 质量靠运气，无法复现
- Agent 越用越乱，没有积累

有了 Harness，你能做到：
- Agent 连续工作数小时，方向不偏
- 质量有保障，错误自动被发现
- 系统越跑越好，经验自动积累

---

## 二、入门前的心态准备：三个必须放弃的旧习惯

**1. 放弃"换更好的模型就能解决问题"的想法**

LangChain 实验证明：同一个模型，只改 Harness，任务通过率从 52.8% 跳到 66.5%。Vercel 删掉 80% 的工具，Agent 反而跑得更快更准。**瓶颈往往不在模型。**

**2. 放弃"写一个超长 prompt 就万事大吉"的想法**

一个塞满规则的大 prompt 会挤掉上下文、让 Agent 模式匹配而非真正理解、而且写完就会立刻腐化。你需要的是**结构**，不是堆字数。

**3. 放弃"一次写好永久有效"的想法**

Manus 6 个月重构了 5 次 Harness，LangChain 1 年内重新架构 3 次。这不是失败，这是正常节奏。**Build to Delete：为删除而构建**，今天的"聪明设计"，明天模型升级后可能就不需要了。

---

## 三、三层基础：从零开始的搭建顺序

Harness 不是一步到位的，按这个顺序来：

```
第一层：Prompt Engineering   ← 你现在可能已经在做了
第二层：Context Engineering  ← 先补这一层
第三层：Harness Engineering  ← 然后才是这一层
```

跳过第二层直接上第三层，Harness 会因为 Agent"什么都不知道"而失效。

### 第一层打好：Prompt 要清晰

这里不展开，你已经熟悉。但有一点常被忽略：**约束语句远比期望语句有效**。

- 差：`"记得写完整的实现"`
- 好：`"不允许出现 TODO 注释，不允许部分实现"`

前者是期望，后者是约束。Agent 对约束的遵守比对期望的理解更可靠。

### 第二层搭好：把知识放进 Repo

这是最容易被跳过、又最关键的一步。**Agent 在运行时看不到的东西，等于不存在。**

你在 Slack 上讨论的决策、你脑子里的隐含规则、Google Docs 里的设计文档——对 Agent 来说全是空白。

**实操步骤：**

```
项目根目录/
├── AGENTS.md          ← 只做目录，约 100 行，告诉 Agent "去哪找什么"
├── ARCHITECTURE.md    ← 项目整体架构，分层规则
└── docs/
    ├── design/        ← 设计决策记录
    ├── product/       ← 产品需求文档
    └── references/    ← 外部库的 llms.txt 等 Agent 友好文档
```

**关键原则**：`AGENTS.md` 是地图，不是百科全书。内容越精简越好，详情链接到 `docs/` 下对应文件。

## 四、三根支柱：Harness 的核心结构

当第二层基础打好，开始搭 Harness 的三根支柱。**初学者建议按顺序来，不要三根同时上。**

---

### 支柱一：评估闭环（最先搭，最重要）

**核心原则：Agent 不能自己给自己打分。**

就像你不会让实习生自己给自己的报告打分一样，Agent 完成任务后总会说"还行"。你需要一个**独立的评估器**。

**小白版实操方案（从简到繁）：**

**阶段 1 - 最简单版（今天就能做）**：写一份验收清单

```markdown
# 验收标准（放在任务描述里）
完成后，请按以下清单自查，每项必须明确回答"是"或"否"：
- [ ] 功能 A 能正常运行（用 X 数据测试）
- [ ] 没有 TODO 或未完成的注释
- [ ] 已有对应的测试用例并通过
```

**阶段 2 - 进阶版**：把验收标准从 prompt 里分离出来

```
task_spec.md       ← 描述"做什么"
eval_contract.md   ← 描述"怎么算做好了"（独立文件）
```

运行时先让 Agent 读 `task_spec.md` 工作，完成后用另一次独立调用加载 `eval_contract.md` 检查产出。两次调用之间没有共享状态——这就是独立性的最简实现。

**阶段 3 - 完整版（Anthropic 方案）**：Planner + Generator + Evaluator 三角色

- Planner 负责把需求扩展为完整 spec
- Generator 按 spec 实现
- Evaluator 用 Playwright/测试框架操作真实应用验证

**何时升级**：当你发现 Agent 频繁"自我说服"（"这个 bug 影响不大"）时，就是升级到阶段 3 的时机。

---

### 支柱二：架构约束（第二搭）

**核心原则：用代码强制执行的规则，远比文档里写的规则有效。**

**阶段 1 - 最简单版**：在 AGENTS.md 里用"禁止语"而非"请求语"

```markdown
# 架构规则（机械执行）
- 禁止在 UI 组件里直接调用数据库
- 禁止在同一个文件里混放业务逻辑和工具函数
- 禁止使用全局变量
```

**阶段 2 - 进阶版**：定义分层架构 + CI 检查脚本

```
数据层 (models/) → 服务层 (services/) → 接口层 (api/) → 视图层 (ui/)
                   单向依赖，不允许反向
```

写一个 `check_arch.py`，在 CI 里跑，违反分层就报错，错误信息写清楚怎么修。

**阶段 3 - 完整版（OpenAI 方案）**：让 Agent 自己生成并维护 linter

当架构规则复杂到手写 linter 很累时，直接让 Agent 根据架构文档生成 linter。错误信息要写成 Agent 能读懂并执行的修复指引。

**立刻能用的小技巧**：减少工具数量。给 Agent 的工具列表，每次都问自己：这个工具真的需要吗？工具越少，Agent 越专注。

---

### 支柱三：记忆治理（有多 Agent 再搭）

**核心问题：单个 Agent 忘事不可怕，多个 Agent 共享一个错误的"经验"才可怕。**

如果你只用单个 Agent，记忆治理先不用管。等你需要多个 Agent 协作时再来处理。

**多 Agent 场景的简单方案**：给共享知识库加一个"验证门"

```
Agent 写入经验 → 标记为"待验证" → 另一次独立调用验证 → 验证通过 → 标记为"已确认"
```

只有"已确认"的知识才被其他 Agent 读取。

---

## 五、熵对抗：系统跑久了怎么防腐化

Agent 系统天然会腐化：文档过期、架构被绕过、知识库堆满过时信息。OpenAI 团队曾每周花 20% 时间手动清理，不可持续。

**解决方案：定期运行一个"清洁 Agent"**

```markdown
# 清洁 Agent 任务
扫描整个 docs/ 目录，找出：
1. 代码里已经删除的功能，但文档还在描述它的
2. AGENTS.md 里指向不存在文件的链接
3. 架构规则被绕过的代码模式

对每个发现，开一个修复 PR，等待人工确认。
```

**"黄金原则"的思路**：与其一遍遍在 prompt 里重复同一条规则，不如把它编码进 linter 或测试里，让它自动运行在每次 Agent 工作上。

## 六、现实生产中的路径图

### 场景 A：个人开发者 / 小项目

**最小可行 Harness**：
1. 建 `AGENTS.md`（100 行以内的地图）+ `docs/` 目录
2. 任务描述里加验收清单（支柱一阶段 1）
3. AGENTS.md 里用禁止语写架构规则（支柱二阶段 1）

**工具推荐**：Claude Code 或 Cursor，不需要额外框架。

### 场景 B：小团队 / 中等复杂度项目

**推荐 Harness**：
1. 完整 docs/ 结构
2. 独立 eval contract 文件（支柱一阶段 2）
3. 分层架构 + CI 检查脚本（支柱二阶段 2）
4. 每周运行一次清洁 Agent

**工具推荐**：LangChain DeepAgents 或 OpenHands。

### 场景 C：多 Agent 并行 / 复杂系统

| 你的痛点 | 对应维度 | 解法重点 |
|---------|---------|---------|
| 单个 Agent 跑久了就偏 | 时间 Scalability | Planner+Generator+Evaluator 三角色 |
| 想让很多 Agent 并行不互相干扰 | 空间 Scalability | 递归 Planner-Worker，Worker 完全隔离 |
| Agent 太多，人来不及管 | 交互 Scalability | ticket 驱动 + Proof of Work 机制 |

**工具推荐**：DeerFlow 2.0。

---

## 七、最容易踩的三个坑

**坑 1：把 AGENTS.md 写成手册** — 超过 300 行就开始腐化，保持简短，细节推到 `docs/`。

**坑 2：用完美主义拖住吞吐量** — 纠错比等待便宜，设定"够好就合并"的标准。

**坑 3：遇到 Agent 失败就去改 Agent** — Agent 卡住是信号，修 Harness，不是修 Agent。

---

## 八、一句话总结

你的工作从**写代码**变成了**设计环境**。环境越好，Agent 越可靠。Agent 越可靠，你的注意力就能放在更高层的决策上。
